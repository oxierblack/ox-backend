# OXIER Backend — التحديثات والنشر على Railway

## 1. الأسواق الحقيقية (Binance)

تم استبدال نظام الأسعار المُحاكاة بالكامل بمصدر حقيقي من Binance، بنفس الأسلوب
الذي يستخدمه ملف الواجهة (`OXIER_FINALL.HTML`):

- `GET /api/v3/exchangeInfo` + `GET /api/v3/ticker/24hr` لجلب كل أزواج `USDT`
  المتاحة فعليًا للتداول، مرتبة حسب حجم التداول.
- نفس منطق تصنيف الفئات (`Gold` / `Forex` / `Crypto`) ونفس روابط أيقونات
  العملات (CoinCap + التجاوزات اليدوية لـ ibb.co).
- يتم مزامنة القائمة كاملة في جدول `MarketSetting`، لكن **فقط أعلى 30 زوج من
  حيث حجم التداول تكون مفعّلة (`isActive: true`) تلقائيًا**. باقي الآلاف من
  الأزواج تُحفظ في القاعدة لكن غير مفعّلة، والأدمن يقدر يفعّل أي زوج إضافي من
  endpoint:

  ```
  PATCH /api/admin/markets/:id   { "isActive": true }
  ```

  > **ليه القائمة محدودة؟** تفعيل كل الـ 2000+ زوج كان سيجعل محرك الإشارات
  > والـ price feed يحسبوا/يبثوا آلاف التحديثات كل ثانيتين لكل عميل متصل —
  > غير عملي فعليًا. 30 زوج هي قابلة للتعديل من `DEFAULT_ACTIVE_COUNT` في
  > `src/services/prices.service.ts`.

- الأسعار نفسها تُحدَّث كل 2 ثانية من `GET /api/v3/ticker/price` وتُبَث عبر
  WebSocket، وتُعاد مزامنة قائمة الأزواج كل ساعة (لرصد عملات جديدة مطروحة على
  Binance أو محذوفة).

## 2. منع الشريك من التسجيل لنفسه تحت وكالته

تم إصلاح المنطق في `src/services/auth.service.ts`. الفحص الجديد يمنع التسجيل
بكود إحالة (`partnerCode`) في حالتين:

1. **نفس الجهاز أو نفس IP الخاص بحساب الشريك نفسه** (كان مسموح به سابقًا لو
   `deviceId` غير مُرسَل في الطلب — وهي الثغرة الأساسية).
2. **نفس الجهاز/IP استُخدم من قبل لتسجيل عميل آخر تحت نفس الشريك** — لمنع
   الشريك من فتح حسابات وهمية متعددة من نفس جهازه.

عند رصد أي من الحالتين، **عملية التسجيل بالكامل تُرفض** برسالة واضحة للمستخدم،
ولا يتم إنشاء حساب على الإطلاق. هذا يتطلب أن يبعث الفرونت إند `deviceId` ثابت
لكل جهاز (fingerprint) في طلب `/api/auth/register`.

⚠️ **ملاحظة مهمة:** هذا الفحص لا يقدر يحل محل التحقق الحقيقي من الهوية —
أي شخص يستخدم متصفح مختلف (incognito) أو يغيّر `deviceId` يدويًا، أو يستخدم
VPN لتغيير الـ IP، يقدر يتجاوزه. هو خط دفاع أول يصعّب الاحتيال البسيط، وليس
حلاً نهائيًا. لو محتاج حماية أقوى، فكر في KYC أو ربط وسيلة الدفع بدل الاعتماد
على device fingerprint فقط.

## 3. النشر على Railway

### المتطلبات قبل النشر
1. قاعدة بيانات MongoDB — الأسهل: [MongoDB Atlas](https://www.mongodb.com/atlas)
   (فيه باقة مجانية كافية للبداية)، أو إضافة MongoDB كـ plugin من داخل Railway
   نفسه.
2. مفتاح [Resend](https://resend.com) لإرسال إيميلات OTP فعليًا (اختياري —
   لو مش موجود، الأكواد تظهر في الـ logs فقط، وهذا غير عملي بعد النشر الفعلي).

### خطوات النشر
1. ارفع محتويات هذا المجلد (`api-server/`) إلى مستودع GitHub خاص بك.
2. من Railway: **New Project → Deploy from GitHub repo**.
3. أضف متغيرات البيئة التالية من تبويب **Variables** (انظر `.env.example`):

   | المتغير | ملاحظات |
   |---|---|
   | `PORT` | Railway يحددها أوتوماتيكيًا، غالبًا لا تحتاج لتعيينها يدويًا |
   | `MONGODB_URI` | إلزامي — رابط الاتصال بقاعدة البيانات |
   | `JWT_SECRET` | إلزامي، 16 حرف على الأقل — ولّده بـ `openssl rand -hex 64` |
   | `ADMIN_EMAIL` / `ADMIN_PASSWORD` | بيانات حساب الأدمن الذي يُنشأ تلقائيًا أول تشغيل |
   | `RESEND_API_KEY` | لإرسال إيميلات OTP فعليًا |
   | `FRONTEND_URL` | لبناء روابط دعوة الشركاء |
   | `BASE_URL` | دومين الباك إند نفسه على Railway، لبناء روابط صور الإيداع |

4. Railway سيتعرف على المشروع تلقائيًا عبر Nixpacks وملف `railway.json`
   المرفق (يبني بـ `npm run build` ويشغّل بـ `npm run start`، مع health check
   على `/api/healthz`).
5. بعد أول نشر، **غيّر كلمة سر الأدمن الافتراضية فورًا** من لوحة الأدمن.

### ⚠️ تنبيه مهم: تخزين صور إثبات التحويل (`/uploads`)

نظام ملفات Railway **مؤقت (ephemeral)** — أي صورة يرفعها العميل كإثبات تحويل
ستُحذف عند أي إعادة نشر (deploy) أو إعادة تشغيل تلقائي للسيرفر. هذا غير مقبول
لمنصة مالية لأن دليل الإيداع نفسه قد يضيع.

**الحل الموصى به:** قبل النشر الفعلي على عملاء حقيقيين، استبدل
`src/services/storage.service.ts` بتخزين خارجي دائم مثل:
- Cloudinary (الأسهل، فيه باقة مجانية)
- AWS S3 / Backblaze B2
- Railway Volume (تخزين دائم مدفوع منفصل عن الكود)

لم يتم تنفيذ هذا التغيير الآن لأنه يتطلب منك اختيار الخدمة وحساب API key لها —
أخبرني أي خدمة تفضل وأجهزها لك.

## 4. مهام كل لوحة (Endpoints)

موجودة بالفعل في الكود الأصلي، منظمة بمسارات منفصلة وواضحة، كل واحدة محمية
بـ middleware يفحص الدور (`role`):

| اللوحة | المسار | الدور المطلوب | أهم المهام |
|---|---|---|---|
| **الأدمن** | `/api/admin/*` | `admin` | إحصائيات المنصة، إدارة المستخدمين والأرصدة، تعليق الحسابات، إدارة الأسواق، تأكيد/رفض المعاملات، إدارة الشركاء وعمولاتهم |
| **المالية (Finance)** | `/api/finance/*` | `finance`, `admin` | مراجعة طلبات الإيداع/السحب المعلّقة، تأكيدها أو رفضها، ملخص يومي |
| **الدعم (Support)** | `/api/support/*` | `support`, `admin` (+ `client` لإنشاء تذاكره) | صندوق وارد التذاكر، الرد على العملاء، إغلاق التذاكر، عرض بيانات العميل (آخر صفقات/معاملات) عند الرد |
| **الشركاء (Partner)** | `/api/partner/*` | `partner`, `admin` | لوحة تحكم الشريك، عملاؤه، عمولاته، طلب سحب العمولة |
| **العميل (Client)** | `/api/trade`, `/api/wallet` | `client` | فتح/إغلاق الصفقات، الإيداع، السحب، عرض المحفظة |

لكل لوحة middleware دور خاص بها (`requireRole(...)`) مطبّق على مستوى الراوتر
بالكامل، فمستخدم بدور `support` مثلاً **لا يقدر** فعليًا يصل لأي endpoint تحت
`/api/admin` ولا العكس — هذا كان موجود ومُطبَّق صحيحًا من الأساس.

## 6. ملاحظات لم تُنفَّذ (تحتاج قرار منك)

- **تخزين صور الإيداع بشكل دائم** (انظر قسم 3) — يحتاج اختيارك لخدمة تخزين.
- **CORS مفتوح بالكامل** (`origin: "*"`) — مقبول مرحليًا أثناء التطوير، لكن
  يُفضّل تقييده لدومين الفرونت إند الفعلي بعد إطلاق الإنتاج.
- **عدم التحقق من صلاحية `userId` عند إنشاء شريك جديد** (`/partner/create`) —
  تحسين بسيط، أخبرني إذا كنت تريده.
